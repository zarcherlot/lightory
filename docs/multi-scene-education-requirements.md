# 多场景教育软件开发需求文档

## 当前结论

更新时间：2026-07-08

下一阶段目标是在当前 Lightory 教育 MVP 上扩展 8-10 个可进入、可编辑、可运行 agent 小队的场景：

- 1 个小镇主场景。
- 2 个小镇周边场景。
- 5-7 个室内主题场景。

技术路线建议：

- [建议] 不引入 Unity / Godot / Phaser 等专业游戏引擎。
- [建议] 不引入大型三方地图编辑器作为运行时编辑器。
- [建议] 扩展当前 `OfficeLayout`、`OfficeCanvas`、`EditorToolbar`、`RoleDock`、`EducationModeOverlay`，做轻量多场景编辑与切换系统。
- [可选] 后续如果像素资产量明显增长，可把 Tiled Map Editor 作为离线资产制作工具，但不作为产品内编辑器。

理由：

- 当前核心体验是 agent 编排、能力卡连接、Play 模式运行、结果卡传递，不是复杂物理或关卡游戏。
- 当前工程已有网格、家具、墙地面、角色、保存布局、角色配置和教育 overlay。
- 引入专业引擎会增加 React / WebView / 服务端消息协议同步成本，反而拖慢教育功能迭代。

## 产品目标

把单一办公室 MVP 扩展为“小镇 + 多主题房间”的学习空间。小朋友可以从小镇进入不同场景，在每个场景里用同一套低门槛交互设计 agent 小队：

- 拖入角色。
- 拖能力卡建立协作关系。
- 配置角色任务。
- 点击运行小队。
- 观察角色处理、输出和卡片传递。
- 后续查看任务回放。

核心原则：

> 场景提供学习主题和任务语境；agent 编排能力在所有场景中保持一致。

## 场景范围

### A. 小镇主场景

用途：

- 作为全局入口和导航 hub。
- 展示小镇地图、主要建筑、任务入口、角色活动状态。
- 让小朋友选择进入哪个场景。

建议命名：

- 小镇中心。

功能：

- [未完成] 显示 8-10 个可进入场景入口。
- [未完成] 点击建筑进入对应场景。
- [未完成] 显示每个场景的主题、推荐任务、完成状态。
- [未完成] 支持回到上次编辑的场景。
- [未完成] 支持从主场景创建新任务。

不要求：

- 不做开放世界自由移动。
- 不做实时 NPC 系统。
- 不做复杂地图寻路任务。

### B. 小镇周边两个场景

#### 1. 小镇公园

主题：

- 天气、出行、自然观察、活动计划。

典型任务：


推荐角色：

- 出行管家。
- 百科老师。
- 总结员。
- 检查员。

- 草地、树、花坛、长椅、路线指示牌、池塘、观察点。

#### 2. 小镇周边探索区

推荐角色：

- 地图导航员。
- 气象学家。
- 出行管家。

## 通用场景能力

每个场景都必须支持当前 MVP 的核心功能：

- [未完成] 场景内可放置角色。
- [未完成] 场景内显示角色能力卡。
- [未完成] 能力卡可拖给其他角色建立连接。
- [未完成] 连接曲线常驻显示，并支持 hover / click 聚焦。
- [未完成] Play 模式隐藏能力卡，显示角色运行状态。
- [未完成] Play 模式显示结果卡摘要。
- [未完成] 结果卡沿连接飞行动画传递。
- [未完成] 缺少输入卡时显示角色气泡提示。
- [未完成] 每个场景有独立任务卡文案。
- [未完成] 每个场景有独立推荐角色、默认角色配置和推荐连接。

跨场景规则：

- 进入任意场景后，agent 编排交互保持一致。
- 切换场景时，当前场景的 layout、角色、连接、任务配置必须保存。
- 场景之间不共享临时运行状态。
- 场景之间可以共享角色定义和角色任务模板。

## 数据模型需求

当前 `OfficeLayout` 只描述一个房间布局。下一阶段需要新增场景层数据。

建议新增：

```ts
interface EducationScene {
  id: string;
  name: string;
  kind: 'town' | 'outdoor' | 'indoor';
  theme: string;
  description: string;
  taskGoal: string;
  layout: OfficeLayout;
  recommendedRoleIds: string[];
  defaultRoleConfigs: Record<string, RoleRuntimeConfig>;
  defaultConnections: EducationConnection[];
  entryPoint?: {
    sceneId: string;
    col: number;
    row: number;
  };
  status?: 'not_started' | 'edited' | 'completed';
}

interface EducationSceneLibrary {
  version: 1;
  activeSceneId: string;
  scenes: EducationScene[];
}
```

需要注意：

- `OfficeLayout` 保持向后兼容，不直接塞入大量产品语义。
- 场景元数据放在新文件或新 store 中，例如 `sceneTemplates.ts`、`educationScenes.ts`。
- 第一阶段可以只在前端保存；后续再接入服务端持久化。
- 旧布局加载时，应自动包装成一个默认场景，例如 `default_office`。

## 资产需求

第一阶段资产策略：

- 使用当前像素家具系统扩展主题素材。
- 主题元素优先做成 furniture catalog 资产，而不是新引擎对象。
- 每个场景先保证 8-15 个主题资产，避免资产量拖慢功能验证。

建议新增素材包：

- `town-core`：建筑入口、公告牌、道路、广场、路灯、喷泉。
- `outdoor-park`：树、花坛、池塘、长椅、观察牌。
- `outdoor-explore`：路牌、公交站、河岸、山坡、安全提示牌。
- `lab-chemistry`：实验台、试管、烧杯、安全柜、护目镜、洗手池。
- `greenhouse`：花盆、育苗架、温度计、浇水壶、植物标签。
- `library`：书架、阅读桌、黑板、资料卡、地球仪。
- `creative-workshop`：画架、颜料、展示板、素材墙、打印机。
- `travel-room`：衣柜、背包、地图桌、天气屏、物品清单板。
- `math-room`：数字白板、图表墙、计时器、算盘。

如果素材需要外部制作：

- 优先保持现有像素风尺寸和透明背景。
- 每个资产必须有 manifest 条目：id、label、category、footprint、orientation、state。
- 可旋转家具仍走现有 `groupId + orientation` 机制。
- 可开关家具仍走现有 `state: on/off` 机制。

## 第三方编辑器 / 游戏引擎评估

### 不建议引入专业游戏引擎

不建议：

- Unity。
- Godot。
- Phaser。
- PixiJS 重写渲染层。

原因：

- 当前不是重交互游戏，而是教育生产力工具。
- React UI、WebView、现有消息协议、角色运行状态已经形成闭环。
- 专业引擎会迫使 OfficeCanvas、编辑器、资产系统、角色 overlay 大量重写。
- 表单、任务卡、配置弹窗、Console、设置面板仍然更适合 React。

### 暂不引入产品内三方地图编辑器

不建议把 Tiled、LDtk 等直接嵌入产品内作为编辑器。

原因：

- 儿童用户需要低门槛场景搭建，不需要专业图层、碰撞、tile set 概念。
- 当前编辑器已经有足够的最小能力。
- 嵌入专业编辑器会破坏产品一致性。

### 可选的离线资产工具

可选：

- Tiled Map Editor：仅用于设计默认小镇底图或批量摆放模板。
- Aseprite / LibreSprite：用于制作像素素材。

接入方式：

- 离线导出 JSON / PNG。
- 通过脚本转换为当前 `OfficeLayout` 和 furniture catalog。
- 不暴露给儿童用户。

## 风险与约束

- 多场景不是简单多张地图，还要保存每个场景的角色、连接和配置。
- 主题资产容易成为瓶颈，第一阶段应优先使用少量高识别度资产。
- 小镇主场景如果做成“自由移动地图”会扩大范围，建议先做点击式 hub。
- 场景任务差异不应导致角色系统分叉；优先复用现有角色定义和任务模板。
- 当前 P7 任务回放仍未完成，多场景阶段要预留回放数据结构，但不强制先做完整回放。

## 不做范围

第一阶段不做：

- 3D 场景。
- 物理碰撞和复杂角色控制。
- 多人在线协作编辑。
- 专业地图图层编辑。
- 完整剧情任务系统。
- 复杂 NPC 对话树。
- 游戏引擎迁移。
