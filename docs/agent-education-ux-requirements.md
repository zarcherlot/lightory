# Agent 教育软件 UX 需求文档

## 产品目标

把当前 Pixel Agents 从“开发者观察 agent 工作状态的像素办公室”，扩展为一个适合小朋友学习 agent 设计与编排的教育软件。

核心学习目标：

- 理解一个 agent 是一个有能力的角色。
- 理解 agent 之间可以传递信息。
- 理解复杂任务可以拆成多个角色协作完成。
- 通过可视化运行过程观察输入、处理、输出、传递。
- 用低门槛交互替代复杂 workflow、DAG、console 操作。

## 核心场景

第一条建议落地场景：

> 明天去学校 / 公园，需要准备什么？

基础角色：

- 天气预报员：查询天气，输出天气卡。
- 穿衣建议员：根据天气卡输出穿衣卡。
- 出行提醒员：根据天气卡输出出行卡。
- 小队队长：收集其他卡片，输出最终准备清单。

目标链路：

```text
天气预报员 -> 穿衣建议员 -> 小队队长
天气预报员 -> 出行提醒员 -> 小队队长
```

## P0：编辑模式与 Play 模式

必须先建立两个清晰模式。

### 编辑模式

用途：让小朋友设计 agent 小队。

表现：

- 角色头顶显示“能力卡”。
- 能力卡表示这个角色可以输出什么。
- 小朋友可以拖拽能力卡到另一个角色身上，建立协作关系。
- 成功后短暂显示一条半透明拱形曲线连接两个角色。
- 曲线一段时间后淡出，不长期占据画面。

示例：

```text
天气预报员头顶：[天气卡]
穿衣建议员头顶：[穿衣卡]
小队队长头顶：[计划卡]
```

### Play 模式

用途：运行任务，观察 agent 工作过程。

表现：

- 隐藏编辑模式下的头顶能力卡。
- 恢复角色原有运行状态表现。
- 天气预报员继续显示天气图标轮播。
- 运行过程中显示真实结果卡，而不是能力卡。
- 信息流动时，可短暂显示结果卡从一个角色飞到另一个角色。

规则：

> 编辑模式显示“我能给别人什么”；Play 模式显示“我正在做什么、我产出了什么”。

## P1：能力卡拖拽建立连接

这是替代“手动画线”的核心交互。

需求：

- 每个角色在编辑模式下头顶显示一张或多张能力卡。
- 能力卡可被拖拽。
- 拖拽时，可接收该卡片的角色高亮。
- 拖到可接收角色身上后建立连接。
- 拖到不可接收角色身上时卡片弹回。
- 连接成功后显示短暂反馈。

成功反馈：

- 目标角色高亮。
- 能力卡吸附一下后回到原位置。
- 两个角色之间出现半透明拱形曲线。
- 曲线上显示卡片类型，例如 `天气卡`。
- 1.5 到 2 秒后曲线淡出。

失败反馈：

```text
我不需要天气卡，可以交给穿衣建议员。
```

或：

```text
穿衣建议员可以接收天气卡。
```

## P2：半透明拱形曲线连接反馈

连接线不常驻，只作为短暂反馈和回放提示。

需求：

- 连接成功时出现。
- 曲线为拱形，连接两个角色。
- 带箭头方向。
- 有透明度，避免干扰画面。
- 颜色跟卡片类型一致。
- 显示 1.5 到 2 秒后淡出。
- 平时隐藏。

额外交互：

- 鼠标悬停某个角色时，可短暂显示该角色相关连接。
- 点击角色时，可显示它的上游和下游连接。
- Play / 回放时，信息流动前可重新显示对应曲线。

## P3：任务卡片与 Play Button

任务执行需要明确的 Play button。

需求：

- 任务卡片上显示任务目标。
- 有明确的 `运行小队` 按钮。
- 点击后进入 Play 模式。
- Play 模式中禁止编辑能力卡连接。
- 运行结束后可回到编辑模式。

状态设计：

```text
未开始：运行小队
运行中：暂停 / 停止
完成：再跑一次 / 查看回放
出错：修一修 / 再试一次
```

原则：

> 拖角色、拖能力卡是搭建机器；按 Play 是让机器跑起来。

## P4：角色配置二级窗口

不要从 Console 输入天气城市。点击角色后弹出角色配置窗口。

以天气预报员为例：

```text
天气预报员

我负责：查询天气

城市：
[ 上海 ]

日期：
[ 明天 v ]

我会输出：
[x] 温度
[x] 是否下雨
[x] 风力
[ ] 空气质量

[保存] [运行这个角色]
```

需求：

- 点击角色打开二级窗口。
- 支持编辑角色参数，例如城市、日期。
- 支持保存配置。
- 保存后角色卡片或面板显示当前配置。
- 天气预报员不再从 Console 读城市。

进阶需求：

- 提供 `简单模式 / Markdown 模式`。
- 简单模式用表单。
- Markdown 模式允许编辑底层任务说明。
- 表单和 Markdown 尽量双向同步。

Markdown 示例：

```md
# 天气预报员任务

请查询以下地点的天气：

- 地点：上海
- 日期：明天

输出：

- 天气情况
- 温度
- 是否下雨
- 风力
```

原则：

> Markdown 可以存在，但不应该是儿童用户的唯一入口。

## P5：能力卡与结果卡区分

需要明确区分两类卡片，避免交互混乱。

### 能力卡

出现于编辑模式。

表示：

```text
这个角色能产出什么
```

例子：

- 天气卡。
- 穿衣卡。
- 出行卡。
- 计划卡。
- 检查卡。

用途：

- 拖给其他角色。
- 建立协作关系。

### 结果卡

出现于 Play 模式。

表示：

```text
这个角色实际产出了什么
```

例子：

```text
上海，明天，小雨，18°C
```

```text
建议：外套 + 雨伞
```

用途：

- 展示真实运行结果。
- 沿已建立连接传递给下一个角色。

## P6：信息流动画

Play 模式下，运行过程要可观察。

需求：

- 角色执行时显示对应状态动画。
- 天气预报员保留天气图标轮播。
- 角色完成后生成结果卡。
- 结果卡沿之前建立的连接关系传递给下一个角色。
- 接收角色收到卡片后进入运行状态。
- 最终由小队队长生成结果清单。

天气流程示例：

```text
天气预报员生成：天气结果卡
天气结果卡飞向：穿衣建议员
穿衣建议员生成：穿衣建议卡
穿衣建议卡飞向：小队队长
小队队长生成：最终准备清单
```

## P7：任务回放

适合教学使用。

需求：

- 任务完成后提供“查看回放”。
- 回放时逐步展示每个角色的输入、处理、输出、传递。
- 可以暂停、继续、重播。
- 回放时可临时显示连接曲线和卡片流动。

回放信息：

```text
天气预报员收到：城市=上海，日期=明天
天气预报员输出：天气卡
穿衣建议员收到：天气卡
穿衣建议员输出：穿衣卡
小队队长收到：穿衣卡、出行卡
小队队长输出：准备清单
```

## P8：错误与推荐机制

错误应该转化为学习提示。

需求：

- 如果角色缺少输入，角色用气泡提示。
- 如果能力卡拖给了不合适的角色，系统推荐合适角色。
- 如果任务无法运行，指出缺少哪个角色或哪张卡。
- 不显示技术错误给儿童用户。

示例：

```text
我还不知道明天的天气。可以请天气预报员先帮我查一下吗？
```

```text
穿衣建议员需要天气卡，请把天气卡交给它。
```

```text
小队队长还没有收到任何建议卡。
```

## P9：角色体系儿童化

当前 Planner / Researcher / Architect / Builder / Reviewer / Tester 偏开发者。

建议新增儿童友好角色体系，不一定立即替换现有角色。

第一批角色：

- 天气预报员。
- 穿衣建议员。
- 出行提醒员。
- 小队队长。

后续可扩展：

- 地图导航员。
- 百科老师。
- 计算小能手。
- 翻译员。
- 故事作家。
- 海报设计师。
- 检查员。
- 总结员。
- 提问员。

## 建议开发顺序

1. P0：编辑模式 / Play 模式基础状态。
2. P1：角色头顶能力卡 + 拖拽到角色建立连接。
3. P2：连接成功后的半透明拱形曲线反馈。
4. P3：任务卡片 Play button 与运行状态切换。
5. P4：点击角色打开配置窗口，天气地点从界面编辑。
6. P5：能力卡 / 结果卡数据结构区分。
7. P6：Play 模式下结果卡传递动画。
8. P8：缺少输入、错误连接的儿童化提示。
9. P7：任务回放。
10. P9：扩展儿童友好角色体系。

## 第一阶段 MVP 范围

建议第一阶段只做天气任务闭环：

- 角色：天气预报员、穿衣建议员、小队队长。
- 编辑模式：头顶能力卡。
- 能力卡拖拽建立连接。
- 连接成功显示短暂拱线。
- 点击天气预报员可配置城市和日期。
- 点击 Play 后隐藏能力卡，恢复天气图标轮播。
- 天气结果卡传给穿衣建议员。
- 穿衣建议卡传给小队队长。
- 输出最终准备清单。

第一阶段目标不是做完整 workflow，而是验证：

> 小朋友是否能通过“拖能力卡”理解 agent 之间的信息传递。
